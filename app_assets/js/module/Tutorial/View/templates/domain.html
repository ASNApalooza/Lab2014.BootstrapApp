Part 1: Domain

Creating a domain model is the first (and most important step) in building a good web application. Since this contacts application is a very simple example, we will not be concerned with real business logic.

Our domain model will only consist of a simple Contact entity, and a Repository interface. A real-world Domain Model should contain the core business logic of an application. With our simple application, we are not implementing Domain Driven Design, so an anemic domain model is not as bad as it would be on a large project.

Why bother? Defining our Contact as a strongly typed data structure provides IntelliSense in Visual Studio and compile-time safety of our data. Creating a Repository interface provides a simple contract that we can use to create, fetch, update, and delete contacts. In part 2 we will see how this lets us change databases without changing most of the code.

The Contact entity starts as a plain-old AVR class: 

Using System

BegClass Contact Access(*Public)

    // TODO: Add properties

EndClass

Each property contains a simple getter and setter: 

DclFld _id Type(*Integer4) Access(*Protected)
BegProp id Type(*Integer4) Access(*Public)
    BegSet
        _id = *PropVal
    EndSet
    BegGet
        LeaveSr _id
    EndGet
EndProp

Each contact needs to support the following values:

  • id (integer)
  • company (string)
  • firstName (string)
  • lastName (string)
  • email (string)
  • phone (string)
  • address1 (string)
  • address2 (string)
  • city (string)
  • state (string)
  • postalCode (string)

Once the Contact class is complete, it should look like this:
